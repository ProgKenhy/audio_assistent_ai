# AI Snake — Техническая документация AI части проекта

---

# 1. Общая идея проекта

Проект представляет собой систему, в которой голос пользователя в реальном времени преобразуется в команды управления игрой. В отличие от классических систем распознавания речи (например, ASR), здесь не требуется распознавать текст — задача сводится к **классификации коротких аудио-команд**.

Это сильно упрощает постановку задачи, но при этом создаёт специфические сложности:

* команды короткие (часто < 1 секунды)
* сильно зависят от конкретного голоса
* легко путаются с шумом
* могут находиться в любой части аудиофрагмента

Поэтому архитектура проекта строится не вокруг "распознавания речи", а вокруг **устойчивой классификации аудио-сигналов в реальном времени**.

---

# 2. Полный pipeline обработки аудио

Вся система работает как последовательность преобразований:

```
аудио сигнал → сегментация → признаки → модель → постобработка → команда
```

Разберём каждый этап подробно.

# 3. Сегментация аудио (VAD + FSM)

## 3.1 Почему нельзя подавать поток напрямую в модель

Если брать поток с микрофона и сразу подавать его в модель, возникают проблемы:

1. модель будет получать шум между словами
2. границы команд будут случайными
3. часть команды может "обрезаться"
4. сильно возрастёт количество ложных срабатываний

Поэтому сначала нужно выделить **целостные фразы**, которые, с высокой вероятностью, содержат одну команду.

---

## 3.2 Voice Activity Detection (VAD)

Используется WebRTC VAD — это классический алгоритм, который определяет, есть ли речь в коротком аудио-фрейме.

Он работает не как нейросеть, а как набор эвристик:

* анализ энергии сигнала
* спектральные характеристики
* статистика шума

Аудио разбивается на маленькие куски по 10 мс. Для каждого такого куска VAD возвращает бинарный ответ: речь / не речь.

Чтобы повысить устойчивость, используется голосование:
если больше половины фреймов внутри чанка содержат речь — считаем, что весь чанк содержит речь.

Это уменьшает влияние случайных шумов.

---

## 3.3 FSM (сборка фразы)

Даже с VAD мы получаем только "есть речь сейчас или нет". Но нам нужна целая команда.

Для этого используется конечный автомат:

1. **WAIT** — ожидание начала речи
2. **SPEAKING** — накопление аудио
3. **SILENCE** — фиксация окончания

Алгоритм:

* пока речи нет — игнорируем сигнал
* как только появляется речь — начинаем запись
* продолжаем запись, пока речь не закончится
* если наступает тишина (несколько подряд чанков без речи) — считаем, что команда закончена

Таким образом, мы получаем один буфер аудио, который примерно соответствует одной команде.

# 4. Feature Engineering (признаки)

Модель не работает напрямую с "сырым" звуком. Вместо этого используется преобразование в MFCC.

## 4.1 Почему не raw audio

Сырые аудио-данные:

* высокоразмерные
* содержат много лишней информации
* плохо обучаются на маленьких датасетах

MFCC — это сжатое представление звука, ориентированное именно на речь.

---

## 4.2 MFCC

MFCC (Mel-Frequency Cepstral Coefficients) извлекаются так:

1. сигнал разбивается на окна (~25 мс)
2. для каждого окна считается спектр (FFT)
3. спектр переводится в mel-шкалу (приближает человеческое восприятие)
4. применяется логарифм
5. применяется DCT

Результат:
матрица размером (n_mfcc, T)

где:

* n_mfcc = 40
* T — количество временных шагов

---

## 4.3 Delta и Delta²

MFCC отражает "состояние" сигнала, но не его динамику.

Чтобы добавить динамику:

* delta — первая производная
* delta² — вторая производная

Итог:

```
[MFCC, delta, delta²] → 120 каналов
```

Это даёт модели информацию:

* как меняется звук
* ускорение изменения

---

## 4.4 Почему это важно

Команды "go" и "stop" могут иметь похожие спектры, но отличаются по динамике произношения. Delta помогает это различать.

---

# 5. Проблема фиксированной длины

Модель ожидает вход фиксированного размера (100 фреймов), но реальные команды имеют разную длину.

Если просто:

* обрезать → теряется часть команды
* дополнять нулями → команда может оказаться в начале/конце

Это приводит к нестабильности.

---

# 6. Sliding Window

Sliding window решает проблему положения команды внутри аудио.

Вместо одного окна берётся несколько:

```
окно 1: [0:100]
окно 2: [20:120]
окно 3: [40:140]
...
```

Каждое окно проходит через модель.

---

## Почему это работает

Если команда попала:

* в начало → её поймает первое окно
* в середину → второе
* в конец → последнее

В результате вероятность команды усредняется по всем окнам.

---

# 7. Test-Time Augmentation (TTA)

Дополнительно каждое окно слегка сдвигается (на 1–2 фрейма).

Это аналог аугментации, но во время инференса.

Почему это полезно:

* модель становится менее чувствительной к точному положению сигнала
* уменьшается variance предсказаний

Итог:
несколько предсказаний → усреднение → более стабильный результат

---

# 8. Модель: сравнение AudioCNN vs MobileNet1D

Это ключевая часть проекта.

---

## 8.1 AudioCNN (старая модель)

Работает с 2D входом:

```
(каналы=1, высота=120, ширина=100)
```

Использует обычные 2D свёртки.

### Особенности:

* воспринимает MFCC как изображение
* обрабатывает и временную, и частотную ось одинаково

### Проблемы:

1. **Не учитывает природу данных**
   Время и частота — разные оси, но 2D CNN обрабатывает их одинаково.

2. **Избыточные вычисления**
   2D свёртки дороже, чем 1D

3. **Хуже ловит временные зависимости**
   Для аудио важна последовательность, а не "картинка"

---

## 8.2 MobileNet1D (текущая модель)

Работает с входом:

```
(B, 120, 100)
```

Где:

* 120 — каналы (MFCC + delta + delta²)
* 100 — время

---

## 8.3 Почему 1D лучше

В 1D модели:

* свёртка идёт только по времени
* каналы обрабатываются отдельно

Это соответствует природе аудио:

* каналы = признаки
* ось времени = последовательность

---

## 8.4 Depthwise Separable Convolution

Обычная Conv:

* перемешивает каналы и пространство одновременно

Depthwise Separable:

1. depthwise — отдельно по каждому каналу
2. pointwise — объединяет каналы

Преимущества:

* меньше параметров
* быстрее
* меньше переобучение

---

## 8.5 SEBlock

SEBlock (Squeeze-and-Excitation):

1. усредняет сигнал по времени
2. получает вектор важности каналов
3. масштабирует каналы

Это позволяет модели:

* усиливать важные MFCC
* подавлять шумные

---

## 8.6 Residual connections

Добавляется shortcut:

```
output = block(x) + x
```

Это:

* улучшает градиенты
* стабилизирует обучение

---

## 8.7 Итог сравнения

| Характеристика | AudioCNN | MobileNet1D |
| -------------- | -------- | ----------- |
| Тип            | 2D       | 1D          |
| Учёт времени   | слабый   | сильный     |
| Скорость       | ниже     | выше        |
| Параметры      | больше   | меньше      |
| Устойчивость   | ниже     | выше        |

# 9. kNN VoiceAdapter (персонализация)

## 9.1 Проблема

Модель обучена на общем датасете:

* разные голоса
* разные акценты

Но пользователь один.

---

## 9.2 Идея

Добавить слой адаптации:

* запоминать примеры пользователя
* использовать их при предсказании

---

## 9.3 Как получаем embedding

Берётся вход последнего Linear слоя:

* размерность 256
* это "сжатое представление" команды

---

## 9.4 Как работает kNN

1. сохраняем embeddings + метки
2. при новом предсказании:

   * считаем расстояние до всех сохранённых
   * берём k ближайших
   * строим распределение

---

## 9.5 Комбинация с CNN

Итоговая вероятность:

```
P = (1 - α) * CNN + α * kNN
```

где:

* CNN — глобальная модель
* kNN — персональная адаптация

---

## 9.6 Почему это работает

Если пользователь произносит команду нестандартно:

* CNN может ошибаться
* kNN "подтягивает" результат

# 10. Логика принятия решения

Модель выдаёт вероятности для 5 классов.

Но команда принимается только если:

```
confidence ≥ threshold
```

Почему:

* снижает ложные срабатывания
* стабилизирует управление

# Заключение

Проект представляет собой полноценную систему:

* потоковая обработка аудио
* извлечение признаков
* нейросетевая модель
* адаптация под пользователя
* интеграция в игру

Главная ценность проекта — не игра, а архитектура обработки аудио в реальном времени и грамотная комбинация методов (VAD, sliding window, TTA, kNN).
